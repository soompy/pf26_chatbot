## 전문 스크립트

---

### [0:00 – 0:30] 오프닝

**"AI 챗봇은 일반 웹 서비스와 근본적으로 다릅니다."**

버튼 클릭 → 응답 대기 → 결과 표시, 이 패턴이 일반 웹이라면,
AI 챗봇은 응답이 **흘러오는** 서비스입니다.

처음에는 단순히 ChatGPT처럼 생기게 만들려 했습니다. 그런데 설계를 시작하자마자
질문이 생겼습니다.

*"스트리밍 중에 사용자가 다시 전송을 누르면? 네트워크가 끊기면? 응답이 너무 느리면?"*

이 질문들에 답하면서 이 프로젝트가 만들어졌습니다.

---

### [0:30 – 1:10] 핵심 설계 결정 ① — AI 응답 상태 모델링

AI 응답에는 일반 UI에 없는 단계가 있습니다.
**thinking → streaming → done**, 그리고 언제든 **error**가 발생할 수 있습니다.

저는 이걸 하나의 상태로 합치지 않고, **5단계 상태 머신**으로 분리했습니다.

```
idle → loading → streaming → done
                           → error
```

그리고 각 단계마다 전용 컴포넌트를 만들었습니다.

- `ThinkingIndicator` — AI가 추론 중임을 amber dots로 표현
- `StreamingText` — 청크가 쌓이는 동안 단락 단위로 fade-in
- `MessageSkeleton` — 첫 청크 도착 전 shimmer 로딩
- `ChatErrorBanner` — 에러 종류별 다른 색상과 행동 유도

**왜 이렇게 나눴냐면**, 합쳐 놓으면 조건문이 컴포넌트 안에 계속 쌓입니다.
나중에 "thinking 단계에서만 특정 UI를 추가해줘" 라는 요청이 와도
컴포넌트 전체를 건드리지 않아도 됩니다.

---

### [1:10 – 1:50] 핵심 설계 결정 ② — 스트리밍 상태 관리

스트리밍을 처음 구현할 때 가장 고민한 건 **낙관적 업데이트**였습니다.

사용자가 메시지를 보내면, API 응답을 기다리지 않고 즉시 화면에 반영합니다.
그리고 동시에 AI 응답 자리에 빈 플레이스홀더를 먼저 넣어둡니다.
첫 청크가 도착하는 순간 스켈레톤이 사라지고 텍스트가 흐르기 시작합니다.

두 번째 고민은 **취소 처리**였습니다.
`AbortController` 하나로 두 가지를 동시에 해결했습니다.
— 사용자가 중단 버튼을 누르면 즉시 abort
— 30초가 지나면 타임아웃으로 자동 abort

그런데 여기서 한 가지 문제가 있었습니다.
둘 다 `DOMException AbortError`로 잡히는데, **왜 abort가 됐는지**를 구분해야
사용자에게 다른 메시지를 보여줄 수 있습니다.

해결책은 `isTimeoutRef` 플래그를 따로 두고,
catch 블록에서 이 플래그를 먼저 확인하는 것이었습니다.

```typescript
const chatErr = isTimeoutRef.current
  ? new TimeoutError(30_000)   // 타임아웃
  : toChatError(err)           // 사용자 취소 또는 다른 에러
```

---

### [1:50 – 2:30] 핵심 설계 결정 ③ — 에러 처리와 UX

에러 처리에서 가장 중요하게 생각한 건 **"이 에러를 다시 시도해도 되는가"** 였습니다.

네트워크가 끊겼다면? 잠깐 기다렸다 재시도하면 됩니다.
API 키가 잘못됐다면? 재시도해봤자 의미가 없습니다.

그래서 `retryable` 플래그를 에러 클래스에 넣었습니다.
`ChatErrorBanner`는 이 플래그만 보고 재시도 버튼 노출 여부를 결정합니다.

개발자가 매번 "이 에러는 재시도 가능한가?" 를 판단할 필요가 없습니다.
서버 HTTP 상태코드를 보고 `ModelError`가 자동으로 결정합니다.
429는 재시도 가능, 401은 불가능, 5xx는 가능.

그리고 에러 메시지는 **기술 용어를 쓰지 않는 것**을 원칙으로 했습니다.
"HTTP 503" 대신 "AI 서버에 일시적인 문제가 발생했습니다."
"AbortError" 대신 "응답 시간이 초과됐습니다."

사용자는 상태코드를 알 필요가 없습니다.
**다음에 무엇을 해야 하는지**가 중요합니다.

---

### [2:30 – 3:00] 마무리

이 프로젝트를 통해 확인한 것은,
**AI 서비스는 처음부터 AI 응답의 특수성을 고려한 설계가 필요하다**는 점입니다.

나중에 스트리밍을 붙이거나, 나중에 에러 처리를 추가하면
컴포넌트와 상태 관리가 복잡해집니다.

13개의 컴포넌트, 5단계 에러 계층, 상태 머신 기반의 스트리밍 훅—
이 모든 것이 "AI 응답을 어떻게 UX로 번역할 것인가"라는 질문에서 시작됐습니다.

감사합니다.

## 예상 질문 & 답변

### Q. Zustand를 선택한 이유는?

대화 이력을 localStorage에 영속화해야 했고, `persist` 미들웨어가 내장된 Zustand가 적합했습니다.
다만 SSR에서 hydration 불일치 문제가 있어 `skipHydration: true`를 설정하고,
클라이언트 마운트 후 `StoreHydration` 컴포넌트에서 `rehydrate()`를 호출하는 방식으로 해결했습니다.

### Q. Context API 대신 Zustand를 쓴 이유는?

Context API는 값이 바뀌면 해당 Context를 구독하는 컴포넌트 전체가 리렌더됩니다.
스트리밍 중에는 매 청크마다 `content`가 업데이트되는데,
Zustand의 `selector` 기반 구독으로 실제로 필요한 슬라이스만 변경될 때 리렌더를 제한했습니다.

### Q. OpenAI와 Anthropic을 동시에 지원하는 방법은?

두 API의 SSE 이벤트 포맷이 다릅니다.
Anthropic의 `content_block_delta` 이벤트를 서버 사이드 `TransformStream`으로
OpenAI 포맷(`choices[0].delta.content`)으로 변환해 클라이언트 파서를 단일화했습니다.
클라이언트는 모델에 관계없이 동일한 `streamChatCompletion()` 함수 하나로 처리합니다.

### Q. 스트리밍 중 성능 최적화는?

매 청크마다 전체 메시지 목록이 리렌더되는 것을 방지하기 위해
`streamingMessageId`를 별도 상태로 분리해 스트리밍 중인 메시지만 업데이트합니다.
`MessageList`는 `streamingMessageId`가 바뀔 때만, 개별 `MessageBubble`은
자신의 ID가 `streamingMessageId`와 일치할 때만 리렌더됩니다.

### Q. 접근성은 어떻게 고려했나요?

스트리밍 텍스트 영역에 `aria-live="polite"`, `aria-busy="true"` 를 적용해
스크린리더가 응답 생성 중임을 인지하고 완료 후 내용을 읽을 수 있게 했습니다.
에러 배너는 `role="alert"`, 중단 버튼은 상태에 따른 `aria-label`을 명시했습니다.
키보드만으로 전체 플로우(입력 → 전송 → 중단)가 가능합니다.

## 발표 시 주의사항

- **"구현했습니다"보다 "왜 이렇게 결정했는지"** 를 중심으로 말할 것
- 코드를 보여줄 때는 파일 전체보다 **핵심 10줄**을 집중 조명할 것
- 면접관이 끊고 질문하면 현재 파트를 마무리하고 바로 답변할 것
- 디자인 시스템 쇼케이스 페이지(`/design-system`)를 미리 열어두고 시작할 것
